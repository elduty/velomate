## 2. Architecture & Design

### Improvements Since Last Audit

- **Komoot removed cleanly**: `ingestor/komoot.py` and `veloai/komoot.py` both deleted. Route planner no longer uploads to Komoot. GPX output is now the primary artifact. This eliminates the undocumented API dependency (old RP3) and the duplicated `CYCLING_SPORTS` constants.
- **Route planner decomposed**: While `plan()` is still a single long function (~150 lines of orchestration), the actual work is now delegated to well-separated modules: `route_intelligence.py` (data gathering), `route_generator.py` (Valhalla routing), `map_preview.py` (HTML output). This is a significant improvement over the "god function" called out in the previous audit.

### Remaining Issues

| Issue | Severity | Details |
|-------|----------|---------|
| **`plan()` still orchestrates too much** | P2 | `route_planner.py:plan()` handles: input parsing, DB queries, weather, intelligence, Valhalla routing, surface verification, scenic scoring, elevation, trails, safety, map preview, and output formatting. Should be split into a data-gathering phase and a formatting phase at minimum. |
| **Two `db.py` modules** | P2 | `ingestor/db.py` (write-heavy, Docker) and `veloai/db.py` (read-only, local CLI) still share no code. Schema knowledge is still duplicated. Acceptable for the architecture but noted for completeness. |

## 5. Data Integrity

### TSS Calculations

Unchanged from previous audit. Formulas are correct. The NP approximation (using `avg_power` as proxy) still under-estimates TSS for variable efforts.

### FTP Estimation

The rolling window issue (DI1) remains open. `ROWS BETWEEN 1199 PRECEDING AND CURRENT ROW` assumes 1 row = 1 second. Stream gaps inflate the window.

### Dedup Logic

Significantly improved:

1. **Stream preservation** — The intent of the fix (reassign streams to -1, then reattach) is correct, but the implementation is broken because `activity_id = -1` violates the FK constraint (see B10).
2. **Short ride protection** — `find_duplicate_by_distance` now skips rides < 5km (fixes DD2).
3. **Data richness scoring** — `_data_richness()` is a clean function. The `merge_activity_data` logic is sound: richer record wins, gaps filled from loser.

### CTL/ATL/TSB

EMA implementation remains correct. Rest-day decay works properly.

### N+1 TSS Update

`recalculate_fitness` still runs individual `UPDATE activities SET tss = %s WHERE id = %s` per activity (`fitness.py:101-104`). Should use `execute_values` or a single UPDATE with CASE. Carried over from previous audit.
