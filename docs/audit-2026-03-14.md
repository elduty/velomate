# VeloAI Technical Audit — 2026-03-14

## 1. Architecture & Design

The overall architecture is clean and well-separated: **ingestor** (Dockerized polling service) and **veloai** (CLI package) share no code and communicate only through PostgreSQL. This is a good boundary.

### Strengths
- Clear data flow: Strava/Komoot → ingestor → PostgreSQL → CLI/Grafana
- Schema-in-code approach (no migration tool) is appropriate for a single-user self-hosted project
- Docker Compose service dependencies use health checks properly

### Issues

| Issue | Severity | Details |
|-------|----------|---------|
| **Two separate `db.py` modules with no shared code** | P2 | `ingestor/db.py` and `veloai/db.py` both connect to PostgreSQL but use different connection patterns (env var `DATABASE_URL` vs YAML config). This is intentional (ingestor = Docker, CLI = local) but means schema knowledge is duplicated. If a column is added to `activities`, the CLI's `get_routes()` query won't break, but the mental model is fragile. |
| **Two separate `komoot.py` modules** | P2 | `ingestor/komoot.py` handles recorded tour ingestion; `veloai/komoot.py` is a simpler API wrapper for the CLI. Different auth patterns — ingestor uses env vars, CLI uses config. Reasonable separation, but the sport type constants are duplicated (`CYCLING_SPORTS` in both files with slightly different sets). |
| **`route_planner.py` is a god function** | P1 | `plan()` at ~120 lines does: parse duration, resolve date, connect DB, estimate distance, fetch weather, geocode waypoints OR generate smart waypoints, call Valhalla, verify surface, generate preview, upload to Komoot, format output. This should be decomposed into a pipeline. |
| **No shared constants** | P2 | Sport types, surface multipliers, and device priorities are scattered across files with no central definition. |

### Error Handling Gaps

| Issue | File | Details |
|-------|------|---------|
| Silent failures | `veloai/route_planner.py:plan()` | Multiple `try/except Exception: pass` blocks silently swallow errors for weather, surface check, wind analysis, best ride hours. At minimum these should log to stderr. |
| No timeout on Overpass | `route_intelligence.py:verify_surface` | The Overpass query can be slow (10-20s), and if it times out, the 15s timeout on `requests.post` may not be enough for the server to process. No retry logic. |
| Komoot login failures not cached | `veloai/komoot.py` | Every call to `fetch_tours()` attempts login. If credentials are wrong, it fails silently every time. |

### Type Safety

- Python 3.10+ union syntax (`X | None`) is used throughout, which is fine per requirements.txt target.
- No type hints on most function return values in `ingestor/` modules.
- `veloai/route_planner.py:estimate_distance` — `avg_speed` parameter typed as `object`, should be `float | None`.

## 3. New Modules Deep-Dive

### `route_intelligence.py` (443 lines)

**Purpose:** Smart waypoint selection using OSM POIs, Strava segments, Komoot highlights, and ride history density.

**Findings:**

| ID | Issue | Severity |
|----|-------|----------|
| RI1 | **`get_ride_density` SQL injection-safe but broken** — `interval '%s days'` won't work with psycopg2 parameterized queries (see B7 above). This will crash at runtime. | **P0** |
| RI2 | **Komoot tile API unlikely to work** — `get_komoot_highlights()` fetches vector tiles from `api.main.komoot.net` and tries to decode them with `mapbox_vector_tile`. The tile format likely doesn't contain `lat`/`lng` in properties — MVT tiles use tile-local coordinates. The function will return empty results in practice. | P1 |
| RI3 | **Distance calculation uses Euclidean approximation** — `math.sqrt((poi["lat"] - lat) ** 2 + (poi["lng"] - lng) ** 2) * 111` treats lat/lng as equal-scale, which is wrong at non-equatorial latitudes. At Lisbon's latitude (~38.7°N), longitude degrees are ~87km, not 111km. Error is ~21%. | P1 |
| RI4 | **Angle-based dedup is weak** — `atan2(lat - center_lat, lng - center_lng)` checks angular separation, but the threshold of 45° in a 3-waypoint max means at most 8 candidates can be considered. If the best POIs cluster in one direction (e.g., coast to the west), good candidates are discarded. | P2 |
| RI5 | **No rate limiting on Overpass API** — `get_pois()` and `verify_surface()` both hit Overpass with no backoff. Overpass has strict rate limits and will ban IPs that make too many requests. | P1 |
| RI6 | **Strava segment `athlete_count` normalization** — `min(1.0, seg["athlete_count"] / 500)` is an arbitrary threshold. In rural Portugal, popular segments might have 50-100 athletes, meaning all segments score < 0.2. In cycling-dense Netherlands, everything scores 1.0. Should be relative to the local maximum. | P2 |
| RI7 | **`verify_surface` counts road WAYS, not road LENGTH** — Each Overpass element is a way, but ways vary wildly in length. A 50m cobblestone driveway and a 5km asphalt road each count as 1. The surface percentage is meaningless. | P1 |

### `route_planner.py` (~300 lines)

**Purpose:** End-to-end route planning pipeline.

**Findings:**

| ID | Issue | Severity |
|----|-------|----------|
| RP1 | **Wind analysis bearing calculation is oversimplified** — `_analyze_wind` uses `atan2(dlng, dlat)` for bearing, which is not a proper great-circle bearing calculation. Should use proper haversine bearing formula. For short segments this is acceptable, but combined with the Euclidean distance issue (RI3), wind analysis accuracy is questionable. | P2 |
| RP2 | **`_get_strava_token` duplicates OAuth logic** — The ingestor's `strava.py` has its own token refresh. The CLI's `route_planner.py` has a separate implementation. Neither shares code with the other. | P2 |
| RP3 | **`_upload_to_komoot` uses internal Komoot API** — Posts directly to `api.komoot.de/v007/tours/` with raw GPX data. This is an undocumented API endpoint. The auth mechanism (user_id + token from komPYoot login) could break at any time. | P1 (inherent, documented) |
| RP4 | **Fitness adjustment only reduces distance for fatigued riders** — `adjust_for_fitness` multiplies by 0.8 when TSB < -10, but doesn't increase distance when fresh. The "fresh" note says "good day to push" but doesn't actually change the target. Inconsistent UX. | P2 |
| RP5 | **Weather `best_ride_hours` called twice** — In `plan()`, `best_ride_hours` is called once for wind analysis (~line 265) and again for best time suggestion (~line 282). Redundant API processing. | P2 |

## 4. Data Integrity

### TSS Calculation

The formulas are correct:
- **Power TSS:** `(duration_s × NP × IF) / (FTP × 3600) × 100` — implementation uses `avg_power` as a proxy for NP (Normalized Power). This under-estimates TSS for variable efforts. Standard practice when NP isn't available, but worth documenting.
- **HR TSS:** `(duration_h) × (HR_ratio)² × 100` — correct hrTSS formula (Coggan).

| ID | Issue | Severity |
|----|-------|----------|
| DI1 | **FTP estimation rolling window is not truly "rolling 20-min"** — The SQL in `estimate_ftp()` uses `ROWS BETWEEN 1199 PRECEDING AND CURRENT ROW`. This assumes 1 row = 1 second (time_offset increments by 1). If stream data has gaps (common when Strava drops data points), the window covers MORE than 20 minutes, inflating the FTP estimate. | P1 |
| DI2 | **Threshold HR auto-estimation uses 95th percentile of max_hr** — This gives LTHR (Lactate Threshold HR), which is the right metric for hrTSS. Correct. | — |
| DI3 | **CTL/ATL EMA implementation is correct** — `CTL = CTL_prev × (1 - 1/42) + TSS × (1/42)` matches standard Banister impulse-response model. ✓ | — |
| DI4 | **Rest days get CTL/ATL decay but no TSS=0 row** — The `recalculate_fitness` function walks every day from first to last activity, applying `tss = daily_tss.get(current, 0)`. Rest days correctly get TSS=0, so CTL/ATL decay properly. ✓ | — |
| DI5 | **Per-activity TSS is written to DB then immediately re-read** — `recalculate_fitness` first UPDATEs each activity's `tss` column in a loop (N queries), then reads them all back with a single SELECT. The UPDATE loop is an N+1 pattern — could be a single UPDATE with CASE or a temp table. | P2 |

### Deduplication

Two dedup strategies, both reasonable:

1. **Time-window + duration** (`find_duplicate`): matches activities starting within 5 minutes with similar duration (±300s). Used for cross-device dedup (Karoo + Watch recording same ride).
2. **Same-day + distance** (`find_duplicate_by_distance`): matches activities on the same calendar day with distance within ±10%. Used for Strava-Komoot linking.

| ID | Issue | Severity |
|----|-------|----------|
| DD1 | **Merge deletes the existing activity then inserts** — `upsert_activity` does `DELETE FROM activities WHERE id = %s` then inserts. This cascades to `activity_streams` (ON DELETE CASCADE), permanently destroying stream data from the deleted record. If the new record has no streams (e.g., Komoot activity), the merged result loses all stream data. | **P0** |
| DD2 | **Distance-based dedup is too loose for short rides** — ±10% of a 5km ride is ±500m. Two different 5km commutes on the same day would be falsely matched. No minimum distance threshold in `find_duplicate_by_distance`. | P1 |
| DD3 | **Dedup priority comment says "zwift > gps/outdoor > watch"** — but the code has `device_priority = {"karoo": 4, "zwift": 3, "unknown": 2, "watch": 1}`. Karoo (GPS outdoor) has HIGHER priority than Zwift. Comment and code disagree on ordering. Code is likely correct (Karoo has richer data than Zwift). | P2 |

## 7. Performance

| ID | Issue | Severity |
|----|-------|----------|
| P1 | **N+1 TSS update** — `recalculate_fitness` runs individual `UPDATE activities SET tss = %s WHERE id = %s` for each activity. For a year of daily riding (~365 activities), this is 365 individual UPDATEs. Should batch with `execute_values` or a single UPDATE with CASE. | P2 |
| P2 | **Strava sync fetches detail + streams for EVERY activity** — Each activity requires 3 API calls (list item already fetched, then detail, then streams) with 1s + 1.5s sleeps = 2.5s minimum per activity. A 12-month backfill of ~300 activities takes ~12.5 minutes in API calls alone. This is appropriate given Strava rate limits. | — |
| P3 | **`get_ride_density` fetches raw GPS points** — The query `SELECT s.lat, s.lng FROM activity_streams` with a 30-day window could return millions of rows if riding daily with 1-second stream resolution. The `time_offset % 30 = 0` filter reduces this to every 30 seconds, which is reasonable. | P2 |
| P4 | **Overpass query in `verify_surface` generates one `way` clause per sample point** — With 30 sample points, this creates a large union query. Overpass handles this fine for small queries but could be slow. | P2 |
| P5 | **Route intelligence makes 3-4 external API calls sequentially** — OSM Overpass, Strava segments, Komoot tiles (multiple), then ride density DB query. No parallelism. Total latency for `smart_waypoints` could be 10-30 seconds. | P1 |
| P6 | **Connection leaks in `poll_strava()` and `poll_komoot()`** — Each poll call creates a new connection via `_get_healthy_conn()` but never closes it. With autocommit=True and polling every 10 minutes, this leaks ~6 connections/hour. PostgreSQL's default `max_connections` is 100, so it'll crash after ~17 hours. | **P0** |

## 8. Known Limitations Assessment

From CLAUDE.md:

| Limitation | Severity | Mitigation |
|------------|----------|------------|
| **Komoot only exposes recorded rides, not saved/planned routes** | Medium | The new route intelligence module (smart_waypoints) mitigates this by generating fresh waypoints from OSM/Strava data rather than relying on past routes. Good. |
| **All Komoot rides default to name "Ride"** | Medium | Dedup uses distance/elevation buckets, which is reasonable. The distance-based dedup (`find_duplicate_by_distance`) is a better approach than name matching. |
| **komPYoot is an unofficial API wrapper** | High | Both `ingestor/komoot.py` and `veloai/komoot.py` and `route_planner.py:_upload_to_komoot` depend on it. If Komoot changes their API, three code paths break simultaneously. Consider a single Komoot abstraction layer. |
| **Valhalla creates loops mathematically** | Low | The new `route_intelligence.py` adds POI-based waypoints, which is a significant improvement. The generated loops now have meaningful destinations. |
